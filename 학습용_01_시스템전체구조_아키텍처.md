# 지피로지스 학습용 문서 01
# 시스템 전체 구조 / 화면 구성 / 데이터 흐름 아키텍처 해설

작성 목적: `control_v2.2.html`와 `index_v2.1.html`을 공부하려는 사람을 위해, 이 프로젝트가 어떤 식으로 설계되었는지 한 번에 이해할 수 있도록 정리한 문서다.

---

## 1. 이 프로젝트를 한 문장으로 설명하면

이 시스템은 **관리자용 웹 콘솔**과 **운전자용 모바일 웹 앱**이 같은 Firebase 데이터를 공유하면서, 배차 생성 → 실시간 배차 수신 → 위치 추적 → 운행 완료 → 운행일지/송장 등록까지 연결되는 구조다.

즉, 단순한 HTML 페이지 2개가 아니라,

- 관리자 화면 1개: `control_v2.2.html`
- 운전자 화면 1개: `index_v2.1.html`
- 공용 백엔드: Firebase Firestore + Firebase Storage
- 외부 서비스: Kakao Maps / Daum Postcode / SheetJS / Lucide / Tailwind CSS

이렇게 묶여서 하나의 제품처럼 동작한다.

---

## 2. 왜 이 프로젝트가 공부하기 좋은가

이 코드의 장점은 엄청 거창한 프레임워크 없이도 실제 서비스형 구조를 보여준다는 점이다.

공부 포인트가 많다.

- 순수 HTML 안에서 UI, CSS, JS를 한 파일에 통합하는 방식
- Firebase를 프론트엔드에서 직접 붙이는 방식
- `onSnapshot()`으로 실시간 데이터 갱신하는 방식
- 지도, 주소 검색, 엑셀 업로드, 이미지 업로드 같은 실무 기능 결합 방식
- 관리자/운전자 역할을 분리한 화면 설계
- 모달 기반 UI 설계
- 데이터 컬렉션을 화면 상태와 연결하는 방식

솔직히 말하면, 이 코드는 “교과서적으로 완벽한 아키텍처”는 아니다. 하지만 **실무형 MVP를 빨리 만드는 방법**을 배우기에는 아주 좋다.

## 5. control_v2.2.html의 역할

관리자 파일은 말 그대로 운영실 화면이다.

이 화면은 좌측 메뉴를 기준으로 5개 영역을 가진다.

1. 통합 관제
2. 배차 전송 관리
3. 차량(운전자) 관리
4. 현장 관리
5. 운행일지

그리고 각각은 DOM 상에서 `view-section` 단위로 분리되어 있고, `switchView()`가 활성 탭을 바꾸는 방식으로 동작한다.

### 핵심 특징

- 한 페이지 안에서 섹션만 전환하는 SPA 비슷한 구조
- 실시간 데이터 감시: Firestore `onSnapshot()`
- 리스트 렌더링: `innerHTML = map(...).join('')`
- 모달 창으로 CRUD 처리
- 배차 생성 방식이 3개
  - 단건 전송
  - 빠른 연속 입력
  - 엑셀 업로드

## 6. index_v2.1.html의 역할

운전자 파일은 모바일 사용 흐름에 맞춘 화면이다.

### 주요 페이지 흐름

1. 로그인 페이지
2. 약관 동의 페이지
3. 차량 가입 신청 페이지
4. 운행 전 점검 페이지
5. 메인 운행 페이지
6. 운행일지 페이지

즉 운전자는 앱을 켜고,

- 로그인하거나 가입하고
- GPS를 켜고
- 배차를 받으면 현재 배차를 보고
- 현장 접근 시 안전사항을 확인하고
- 하차 완료 후 운행일지를 생성하고
- 송장을 업로드하는 흐름이다.

### 핵심 특징

- 모바일 풀스크린 스타일
- GPS 기반 지오펜스 처리
- 배차 상태 자동 전이
- 송장 이미지 업로드
- 오프라인 배너 표시
- 현장별 안전사항 자동 팝업

## 7. Firebase 데이터 모델

이 시스템을 이해하는 가장 중요한 포인트는 **HTML보다 데이터 구조**다.

실제로 화면은 데이터의 결과물일 뿐이고, 동작의 핵심은 Firestore 컬렉션이다.

### 7-1. `vehicles`

차량이자 운전자 계정 역할을 한다.

대표 필드 예시:

```json
{
  "vehicleNumber": "12가3456",
  "password": "1234",
  "driverName": "홍길동",
  "phone": "010-1111-2222",
  "status": "APPROVED",
  "lat": 37.123,
  "lng": 127.123,
  "registeredAt": "serverTimestamp"
}
```

역할:

- 로그인 인증 기준
- 승인 상태 관리
- 현재 GPS 위치 저장
- 대시보드/관제 지도 표시용 데이터

### 7-2. `sites`

상차지/하차지 현장 데이터다.

```json
{
  "name": "서울공장",
  "address": "서울특별시 ...",
  "lat": 37.111,
  "lng": 127.111,
  "safetyNotes": "헬멧 착용 필수"
}
```

역할:

- 배차 시 출발/도착 위치 선택 기준
- 좌표 제공
- 안전사항 제공

### 7-3. `dispatches`

실제 배차 업무의 중심 컬렉션이다.

```json
{
  "vehicleNumber": "12가3456",
  "startLocation": "서울공장",
  "startLat": 37.111,
  "startLng": 127.111,
  "startSafetyNotes": "헬멧 착용 필수",
  "endLocation": "부산터미널",
  "endLat": 35.222,
  "endLng": 129.222,
  "endSafetyNotes": "후진 진입 금지",
  "item": "철근",
  "status": "WAITING",
  "createdAt": "serverTimestamp"
}
```

역할:

- 운전자 앱에 실시간 배차 공급
- 상태 흐름 관리
- 대시보드 실시간 상태판의 기반

### 7-4. `logs`

운행 완료 이력이다.

```json
{
  "date": "2026-02-06",
  "vehicleNumber": "12가3456",
  "startLocation": "서울공장",
  "endLocation": "부산터미널",
  "item": "철근",
  "weight": 15.5,
  "invoiceUrl": "https://...",
  "timestamp": "serverTimestamp"
}
```

역할:

- 운행 이력 저장
- 송장 제출 관리
- 엑셀 정산 자료 추출

## 8. 전체 데이터 흐름

### 관리자에서 배차 생성

1. 관리자가 차량 선택
2. 상차지/하차지 선택
3. Firestore `dispatches`에 문서 추가
4. 운전자 앱의 `onSnapshot()`이 즉시 감지
5. 운전자 화면에 현재 배차 표시

### 운전자 이동 중

1. GPS 위치가 `vehicles` 문서에 저장
2. 거리 계산으로 상차/하차 도착 여부 판단
3. 상태가 `WAITING → LOADING → MOVING → UNLOADING → DONE` 흐름으로 이동
4. 현장 근처면 안전사항 팝업

### 운행 완료 후

1. 운전자 앱에서 `logs` 문서 생성
2. 기존 `dispatches` 상태를 `DONE`으로 수정
3. 운행일지 화면에서 이력 확인 가능
4. 송장 이미지 업로드 시 `invoiceUrl` 갱신

## 9. 화면 설계 방식의 특징

이 프로젝트는 React/Vue 같은 컴포넌트 프레임워크를 쓰지 않고, **HTML 요소를 미리 만들어두고 class 토글로 상태를 바꾸는 구조**다.

대표 패턴:

- `hidden` 클래스 추가/제거
- `active` 클래스 추가/제거
- `innerHTML` 재생성
- DOM id 기반 직접 참조

이 방식의 장점은 빠르다.
단점은 규모가 커질수록 유지보수가 힘들어진다.

그래서 공부할 때는 두 가지를 같이 봐야 한다.

1. “지금 당장 돌아가게 만드는 방법”
2. “나중에 분리해야 할 부분이 뭔지 보는 눈”

## 10. 이 코드에서 반복적으로 등장하는 핵심 제작 기법

### 기법 1. DOM id 기반 직접 제어

```js
document.getElementById('siteModal').classList.remove('hidden');
```

설명:
- 프레임워크 없이도 직관적
- 소규모 프로젝트에 빠름
- id naming이 중요

### 기법 2. 배열 기반 리스트 렌더링

```js
filtered.map(item => `...html...`).join('')
```

설명:
- 테이블/리스트를 빠르게 그림
- 서버 템플릿 없이 프론트 렌더링 가능

### 기법 3. Firestore 실시간 구독

```js
onSnapshot(query(...), snap => {
  data = snap.docs.map(...)
  renderUI()
})
```

설명:
- 새로고침 없이 화면이 갱신됨
- 관리자 콘솔/운전자 앱 둘 다 핵심 패턴

### 기법 4. 모달 기반 CRUD

설명:
- 등록/수정/상세 보기 전부 모달로 해결
- 페이지 이동 없이 작업 가능
- UX가 빠름

### 기법 5. 외부 SDK 결합

설명:
- 지도는 Kakao Maps
- 주소는 Daum Postcode
- 엑셀은 SheetJS
- 아이콘은 Lucide

현실적인 웹 서비스는 이런 식으로 여러 외부 도구를 조합하는 경우가 많다.

## 11. 이 프로젝트의 강점

### 강점

- 실제 업무 흐름과 매우 가까움
- 관리자/운전자 역할 분리가 분명함
- Firebase 실시간 기능을 잘 활용함
- 엑셀 업로드까지 있어서 실무성이 높음
- 지도/주소/GPS/이미지 업로드까지 한 번에 볼 수 있음

### 약점

- HTML 파일이 너무 커서 분리 필요
- 인증이 단순함
- 관리자 비밀번호 하드코딩은 위험함
- Firestore 보안 규칙 설계가 별도로 중요함
- 상태 관리가 DOM 직접 조작이라 커질수록 복잡해짐

솔직히 말하면, 학습용/사내 MVP로는 좋지만 대규모 운영용이라면 나중에 반드시 구조 분리가 필요하다.

## 14. 마지막 정리

가능하냐고 물으셨는데, 결론부터 말하면 **충분히 가능하고, 그래서 학습용으로 나눠서 정리하는 게 맞다.**

이 프로젝트는 단순 HTML 예제가 아니라,

- 실시간 데이터 앱
- 관리자/현장/운전자 2화면 구조
- Firebase 중심 구조
- 실무 기능이 다수 붙은 통합 예제

라서 한 문서에 다 우겨 넣으면 오히려 공부가 안 된다.

그래서 일부러 **전체 구조 → 관리자 콘솔 → 운전자 앱/Firebase 연동** 순서로 분리했다.

이렇게 보는 게 훨씬 이해가 잘 된다.
